FoodRush - Food Ordering System (100% Comprehensive Context Manual)
================================================================================

This document is intended for AI agents and developers. It serves as an exhaustive reference of the architecture, state management, API contracts, and implementation details of the FoodRush application.

1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
FoodRush is a full-stack monolithic web application (split into frontend and backend directories) designed for food ordering. It features:
- Customer workflows: Browsing restaurants, advanced filtering, cart management, checking out (Stripe/Wallet/COD), tracking orders, and favoriting.
- Admin/Restaurant workflows: Viewing stats, managing menus, and updating order status.

2. TECHNOLOGY STACK & CORE DEPENDENCIES
--------------------------------------------------------------------------------
Backend:
- Node.js & Express.js (v4)
- MongoDB & Mongoose (v8) for ODM
- bcryptjs (v2) for password hashing
- jsonwebtoken (v9) for stateless Auth
- stripe (v22) for payments
- cors, dotenv for configuration

Frontend:
- React (v18)
- Vite (v6) bundler
- axios (v1) for API requests
- Vanilla CSS (No Tailwind/Bootstrap)

4. FRONTEND STATE MANAGEMENT (`FoodOrderApp.jsx`)
--------------------------------------------------------------------------------
The application avoids Redux/Context and stores global state in the root component:
- `view` (String): Manual router. 'home' | 'orders' | 'favorites' | 'profile'.
- `restaurants` (Array): Fetched on mount. Falls back to `MOCK_RESTAURANTS`.
- `search` (String): Text search input.
- `filters` (Object): `{ cuisine: [], priceRange: [], minRating: 0, dietary: [], openNow: boolean }`.
- `sortBy` (String): 'featured' | 'rating' | 'delivery' | 'price'.
- `cart` (Array): Array of added MenuItems with a `qty` and `restaurantId` property.
- `favorites` (Array): Array of saved Restaurant IDs.
- Modal states: `cartOpen`, `checkoutOpen`, `orderSuccess`, `selectedRestaurant`, `selectedMenu`, `selectedReviews`.
- Derived State: `filteredRestaurants` recalculates on every render based on `restaurants`, `search`, `filters`, and `sortBy`.

5. BACKEND API DICTIONARY (REST CONTRACTS)
--------------------------------------------------------------------------------
[AUTH]
- POST `/api/auth/register` : { name, email, password, role } => { token, user }
- POST `/api/auth/login` : { email, password } => { token, user }
- GET `/api/auth/me` (Auth) => { user }

[RESTAURANTS]
- GET `/api/restaurants` (?cuisine, priceRange, rating, dietary, search, sort, page, limit) => { restaurants[], total, pages }
- GET `/api/restaurants/:id` => { restaurant, menu[], reviews[] }
- POST `/api/restaurants` (Admin/RestAuth) : { ...restaurant } => { restaurant }
- PUT `/api/restaurants/:id` (Admin/RestAuth) => { restaurant }

[MENU]
- GET `/api/restaurants/:id/menu` (?dietary, category, maxPrice) => { items[] }
- POST `/api/restaurants/:id/menu` (Admin/RestAuth) : { name, price... } => { item }

[ORDERS]
- POST `/api/orders` (Auth) : { restaurantId, items[{menuItemId, qty}], deliveryAddress, paymentMethod } => { order }
- GET `/api/orders` (Auth) => { orders[] }
- GET `/api/orders/:id` (Auth) => { order }
- PATCH `/api/orders/:id/status` (Admin/RestAuth) : { status } => { order }

[PAYMENTS]
- POST `/api/payments/create-intent` (Auth) : { amount, currency } => { clientSecret }
- POST `/api/payments/wallet/topup` (Auth) : { amount } => { wallet }

[REVIEWS & FAVORITES]
- POST `/api/reviews` (Auth) : { restaurant, rating, body } => { review }
  * Note: Automatically updates `rating` and `reviewCount` on the Restaurant document.
- POST `/api/reviews/:id/helpful` (Auth) => { helpfulCount }
- POST `/api/favorites/:restaurantId` (Auth) => { favorites[] }
- GET `/api/favorites` (Auth) => { favorites[] }

[MISC]
- GET `/api/search?q=` => { restaurants[], menuItems[] }
- GET `/api/admin/stats` (Admin) => { totalUsers, totalOrders, totalRestaurants, totalRevenue }

6. CRITICAL WORKFLOWS & QUIRKS
--------------------------------------------------------------------------------
A. Database Fallback (Resiliency)
   The frontend `useEffect` fetches from `/api/restaurants`. If the API is down or throws a 404, the `catch` block fires and populates `restaurants` with `MOCK_RESTAURANTS`. If a user opens a restaurant via `openRestaurant`, it fetches the menu. If that fails, it falls back to `MOCK_MENU[id]`. This guarantees the UI never breaks.

B. Order Placement Logic
   In `orderController.createOrder`, the system recalculates the price on the server. It queries the DB for each `menuItemId` to get the real `price`, calculates `extraCost` for customizations, adds a 5% `tax` and the restaurant's `deliveryFee`. If `paymentMethod === 'wallet'`, it checks if `req.user.wallet >= total`. It then deducts the wallet and adds `Math.floor(total/10)` as loyalty points.

C. Checkout Mocking
   The frontend `CheckoutModal` builds the payload and calls `/api/orders`. If the user is unauthenticated or the API is unavailable, the `catch` block intercepts the error, but the UI *still* proceeds to show `OrderSuccess` (client-side confirmation) for demonstration purposes.

D. Global CSS Constraints
   The design uses a heavy glassmorphism style, CSS variables for theming (`--primary`, `--surface`), and custom scrollbars. Modals use fixed overlays with `z-index: 1000`.
